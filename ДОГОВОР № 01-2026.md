3.3. Этап 3: Интеграционное тестирование и сверка данных
Срок: в течение 4–7 недель с даты начала работ.
Результат: обработка тестовых данных из всех предусмотренных источников; корректная сверка; подготовленные результаты в согласованных форматах выгрузки.

3.4. Этап 4: Приемо-сдаточные испытания (ПСИ) и приёмка
Срок: в течение 6–8 недель с даты начала работ.
Результат: совместное выполнение контрольных сценариев ПСИ по Программе ПСИ (Приложение № 7); протокол замечаний/доработок (при необходимости); оформление акта сдачи-приёмки.

3.5. Этап 5: Обучение и передача документации
Срок: в течение 7–8 недель с даты начала работ (как правило, после ключевой приёмки либо параллельно с устранением замечаний).
Результат: обучение ключевых пользователей; передача инструкций администратору и пользователям; комплект программной документации.

4. Подтверждение локального on-premise режима
4.1. В рамках ПСИ стороны фиксируют отсутствие передачи данных Заказчика во внешние API/интернет, если такое требование прямо установлено методологией и/или требованиями Заказчика.

5. Риски и зависимости
5.1. Сроки этапов 2–4 зависят от своевременной готовности инфраструктуры Заказчика, включая наличие/доступность GPU (при необходимости) и администраторских ресурсов.
5.2. При изменении условий доступности инфраструктуры Исполнитель и Заказчик согласуют уточнение графика выполнения работ в пределах общего срока по этапу 1.

Приложение № 7 Программа приемо-сдаточных испытаний (ПСИ) «ИИ-Аудитор» v1
1. Общие сведения
1.1. Настоящий документ устанавливает порядок, состав работ и критерии приемки Программы «ИИ-Аудитор» версии 1 (далее — Программа) по результатам приемо-сдаточных испытаний (ПСИ).
1.2. ПСИ выполняются в on-premise контуре Заказчика в изолированной инфраструктуре без передачи данных Заказчика во внешние сервисы/интернет после внедрения Программы (если иное не согласовано сторонами письменно).

1.3. Настоящая Программа ПСИ (Приложение № 7) является документом, определяющим состав испытаний, сценарии ПСИ и критерии приемки. Отдельный чек-лист (как самостоятельное приложение) не требуется: критерии приемки и порядок фиксирования результатов содержатся в настоящем Приложении.

2. Цель и объекты испытаний
2.1. Цель ПСИ: подтвердить соответствие Программы требованиям Технического задания (Приложение № 1) и проверить работоспособность Программы при выполнении согласованных сценариев аудита.
2.2. Объекты испытаний:
а) развертывание и доступность Программы в изолированном контуре Заказчика;
б) корректность загрузки входных данных (УК Excel, отчёт брокера, документы ДЕПО) и их структурирования/парсинга;
в) генерация/формирование правил парсинга (если предусмотрено сценарием) и стабильность формата ответа;
г) сверка данных и формирование результатов аудита;
д) генерация отчетов/выгрузок по сценариям и их соответствие критериям приемки, установленным Программой ПСИ (Приложение № 7) и методологией аудита;
е) отсутствие передачи данных Заказчика во внешние API/интернет в рамках требований Заказчика (в пределах возможностей контурной инфраструктуры и средств контроля).

3. Исходные материалы и обязанности Заказчика
3.1. Исходные материалы для ПСИ, предоставляемые Заказчиком (в объеме и по форме, согласованными сторонами):
а) методология аудита и описание ожидаемых результатов по v1 (в согласованной форме документа);
б) требования к функционалу и допускам в рамках ПСИ в виде матрицы сценариев функциональных проверок v1 (с указанием ID сценариев и критериев приемки «Прошло/Прошло с замечаниями/Не прошло»);
в) тестовые обезличенные данные:
1) **образцы входных файлов**: обезличенные УК (Excel), отчёты брокера (Excel) и документы ДЕПО (PDF или иные предусмотренные форматы), объединенные в наборы, при этом каждый файл/набор помечается ID сценария ПСИ и указывает, является ли набор входными данными для Программы либо эталонным результатом для сравнения;
2) **тестовые выгрузки**: обезличенные ожидаемые результаты обработки по соответствующим ID сценариев ПСИ (эталоны/контрольные примеры), представленные в формате, который подлежит сравнению с результатами Программы (в т.ч. в виде экспортируемых таблиц/файлов или эталонных значений), либо если эталоны не предоставляются — описанные критерии приемки без эталонов;
3) при необходимости — пакеты данных для загрузки (например, архивы, содержащие несколько связанных входных файлов для одного сценария).
3.2. Заказчик обязан обеспечить:
а) наличие изолированной инфраструктуры (выделенный стенд/ВМ) для развертывания и тестирования Программы;
б) сетевые параметры и ограничения no-internet (или эквивалентные ограничения), DNS/URL доступы, при необходимости — сертификаты HTTPS;
в) доступы/учетные записи для следующих категорий пользователей ПСИ:
1) администратор стенда (доступ к развёртыванию/настройке в контуре, к логам сервиса/конфигурации при необходимости для исполнения сценариев);
2) пользователь ПСИ (аудитор/аналитик Заказчика), имеющий доступ к веб-интерфейсу Программы для загрузки входных файлов, запуска сценариев ПСИ и просмотра результатов/выгрузок;
3) при необходимости — пользователь с правами администратора Программы (для управления правилами/параметрами в рамках сценариев);
г) при необходимости — наличие GPU/узла для функционирования локальной LLM и/или возможность ее размещения в контуре.

4. Испытательная среда и условия проведения
4.1. ПСИ проводятся на стенде Заказчика, подготовленном в соответствии с требованиями Приложения № 2 и Приложения № 4.
4.2. Перед началом испытаний стороны фиксируют:
а) факт готовности стенда (акт/уведомление о готовности);
б) версии развернутых компонентов Программы и параметры конфигурации, влияющие на результаты;
в) перечень пользователей/ролей и доступов, необходимых для выполнения сценариев.
г) идентификатор локально используемой LLM (наименование/версия) в составе развертывания для целей ПСИ.

4.3. Критерии готовности к началу ПСИ (совокупно должны быть выполнены):
а) Программа развернута и доступна в изолированном on-premise контуре, а пользователи ПСИ могут выполнить вход в веб-интерфейс и инициировать операции загрузки/запуска сценариев;
б) База данных PostgreSQL доступна, схема/миграции (если предусмотрены Программой) успешно применены;
в) На стенде обеспечены no-internet ограничения (отсутствие внешней передачи данных Заказчика в период ПСИ в пределах требований Заказчика) и применены сетевые/сертификационные параметры;
г) Заказчиком предоставлены и размещены в рабочей зоне все образцы входных файлов и, при наличии, тестовые выгрузки/эталоны, соответствующие перечню ID сценариев ПСИ;
д) определены роли и доступы пользователей ПСИ: администратор стенда, пользователь ПСИ (аудитор/аналитик) и, при необходимости, администратор Программы.

5. Состав и порядок ПСИ
5.1. ПСИ выполняются по сценариям, указанным в Программе ПСИ (далее — «Сценарий ПСИ»). Для каждого Сценария ПСИ в ходе испытаний выполняются следующие действия и фиксируются объективные результаты:

| № | Действие | Ответственный | Описание действия | Результат / критерии приемки |
|---|----------|----------------|--------------------|----------------------------------|
| 1 | Проверка готовности стенда и доступа пользователей | Заказчик совместно с Исполнителем | Проверить, что из учетных записей пользователей ПСИ доступен веб-интерфейс, сервисы в контуре отвечают, а необходимые URL/DNS и HTTPS доступны в рамках no-internet ограничений | Верифицирован факт входа пользователей ПСИ в веб-интерфейс; отсутствуют ошибки подключения/аутентификации; подтверждена доступность необходимых сервисов (фиксируется в ПСИ-протоколе) |
| 2 | Проверка развертывания данных и подключений (PostgreSQL, БД, миграции) | Исполнитель | Проверить корректность конфигурации БД, запуск миграций/инициализации схемы (если предусмотрено), доступность таблиц/схем, влияющих на выполнение Сценариев ПСИ | В ПСИ-протоколе отражено успешное подключение к БД; отсутствие ошибок миграций/инициализации; данные доступны для выполнения Сценариев ПСИ |
| 3 | Подготовка входных данных для Сценариев ПСИ | Заказчик | Предоставить в рабочей зоне стенда образцы входных файлов (УК Excel, отчёты брокера Excel, документы ДЕПО PDF/прочие), помеченные ID Сценариев ПСИ; при необходимости — ожидаемые тестовые выгрузки (эталоны/контрольные примеры) | В рабочей зоне подтверждено наличие всех файлов, соответствующих перечню ID Сценариев ПСИ; обеспечена однозначная прослеживаемость «ID → файлы входа/эталоны» (фиксируется в ПСИ-протоколе) |
| 4 | Выполнение загрузки входных файлов и парсинга | Исполнитель | Запустить Программу в рамках каждого Сценария ПСИ: загрузить соответствующие входные файлы и выполнить извлечение данных по правилам/структурам, используемым Программой в v1 | По каждому Сценарию ПСИ отсутствуют ошибки загрузки/парсинга; получены структурированные результаты парсинга, соответствующие ожидаемым типам/полям, указанным в методологии/Программе ПСИ |
| 5 | Проверка формирования/применения правил парсинга (если предусмотрено сценарием) | Исполнитель | Для Сценариев ПСИ, где предусмотрена интеллектуальная настройка правил: сгенерировать/применить правила и убедиться, что формат ответа/правил соответствует требованиям (например JSON-формат) | Правила и/или ответы валидны по формату и применимы к входным наборам; получены результаты парсинга без форматных ошибок (фиксируется по журналам/выходам) |
| 6 | Выполнение сверки и формирование аудиторских результатов | Исполнитель | Выполнить сверку источников по каждому Сценарию ПСИ, сформировать расхождения/сопоставления согласно методологии аудита | По каждому Сценарию ПСИ результаты сверки соответствуют критериям приемки (допуски/условия совпадения), указанные в методологии/Программе ПСИ; отсутствуют критические несоответствия |
| 7 | Генерация отчетов/выгрузок и проверка состава полей | Исполнитель | Запустить формирование отчетов/выгрузок для каждого Сценария ПСИ и проверить соответствие формату и составу полей ожидаемым результатам/эталонам либо критериям без эталонов | Для каждого Сценария ПСИ сформированы выгрузки/файлы; набор полей и структура соответствуют определенным требованиям; выгрузки пригодны для сравнения с тестовыми выгрузками (если эталоны предоставлены) |
| 8 | Проверка соблюдения on-premise режима и no-internet ограничений | Исполнитель и Заказчик | В рамках средств контроля стенда зафиксировать отсутствие внешней передачи данных/обращения к внешним сервисам в период ПСИ (кроме разрешенного инфраструктурного обмена, если он отдельно согласован) | В ПСИ-протоколе отражено подтверждение отсутствия внешних запросов на обработку данных Заказчика; данные не передавались во внешние API/интернет в период ПСИ (в пределах требований Заказчика) |
| 9 | Оформление ПСИ протокола и результатов по шкале | Исполнитель | Сопоставить по каждому Сценарию ПСИ фактические результаты с критериями приемки и оформить ПСИ протокол | ПСИ-протокол содержит: перечень выполненных Сценариев ПСИ, фактические результаты «Прошло/Прошло с замечаниями/Не прошло», перечень замечаний/доработок (при наличии) и подтверждение достижения критериев приемки |

5.2. Для каждого действия (по каждому Сценарию ПСИ) оформляется результат по шкале:
«Прошло» / «Не прошло» / «Прошло с замечаниями» (с перечнем доработок при необходимости).

6. Критерии приемки
6.1. ПСИ считаются пройденными, если по всем обязательным Сценариям ПСИ, предусмотренным Программой ПСИ (Приложение № 7), получены результаты «Прошло» или «Прошло с замечаниями» с согласованным планом устранения замечаний.
6.2. Критерии «Не прошло» применяются при выявлении:
а) отказа функций, критичных для загрузки/парсинга/сверки/выгрузки по сценариям;
б) передачи данных Заказчика во внешние сервисы/интернет вопреки требованиям;
в) систематической некорректности результатов при воспроизведении на тестовых наборах.

7. Документирование результатов ПСИ
7.1. По итогам ПСИ Исполнитель представляет Заказчику:
а) ПСИ протокол (перечень выполненных проверок, результаты, ссылки на сценарии);
б) при наличии замечаний — перечень доработок и предложенный порядок их устранения;
в) подтверждение соответствия выполненных работ Программе ПСИ (Приложение № 7).
7.2. Итоговые документы ПСИ являются основанием для оформления актов сдачи-приемки и итоговой приемки работ по Договору.
