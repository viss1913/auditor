---

## 1. Назначение системы

Универсальная рабочая среда для аналитиков и аудиторов. Пользователь загружает данные из разнородных источников (Excel, PDF, CSV, внешние БД), общается с AI-ассистентом на естественном языке, и AI помогает:

- Разобрать (спарсить) загруженные файлы
- Подготовить и трансформировать данные в нужный формат
- Сравнить несколько источников данных между собой (аудит)
- Найти расхождения и сформировать отчёт

Система должна быть **универсальной** — не привязанной к одному конкретному сценарию. Конкретные бизнес-задачи (например, сверка сделок ОПИФ) реализуются как настраиваемые правила, а не как хардкод.

## 4. Функциональные модули

### 4.1. Проекты (рабочие пространства)

Вся работа ведётся в контексте проекта. Проект объединяет:
- Источники данных (загруженные файлы, подключения к БД)
- Правила парсинга
- Правила аудита
- Результаты аудита
- Историю диалога с AI

**Пример:**
```
Проект "Аудит ОПИФ Фонд X за 2025"
├── Источники: УК (Excel), Брокер (Excel), ДЕПО (PDF)
├── Правила парсинга: для каждого источника
├── Аудит: сверка УК↔Брокер, сверка УК↔ДЕПО
└── Чат с AI: история вопросов и ответов
```

---

### 4.2. Загрузка и подключение данных

#### Поддерживаемые источники

| Источник | Формат | Описание |
|----------|--------|----------|
| Excel-файлы | .xls, .xlsx, .xlsm | Загрузка файлов, выбор листа, предпросмотр |
| PDF | .pdf (машиночитаемый) | Извлечение текста, парсинг по правилам |
| CSV / TSV | .csv, .tsv | Загрузка с выбором разделителя и кодировки |
| Папка с файлами | Массовая загрузка | Фильтрация по маске имени файла |
| Внешняя БД | PostgreSQL, MySQL, MS SQL | Подключение по credentials, выбор таблицы |

#### Процесс загрузки

1. Пользователь выбирает файл(ы) или вводит параметры подключения к БД
2. Система показывает **предпросмотр** — первые 20–50 строк, названия колонок
3. Пользователь (или AI) задаёт **маппинг колонок** — какая колонка файла соответствует какому полю
4. Пользователь выбирает режим: «Заменить данные» или «Добавить к текущим»
5. Данные парсятся и сохраняются в БД

### 4.3. Движок парсинга

Универсальный парсер, настраиваемый **JSON-правилами**. Правило описывает:
- Какие строки пропустить (шапка)
- Где искать дату, название, суммы, количество и т.д. (карта колонок)
- Условия фильтрации (например: «дебет начинается с 58.01, кредит с 76»)
- Как обрабатывать многострочные записи (когда одна запись занимает 2+ строки)

#### Пример правила парсинга

```json
{
  "name": "Карточка счёта УК — покупка ЦБ",
  "source_type": "excel",
  "skip_rows": 7,
  "column_map": {
    "date": 0,
    "debit_account": 6,
    "credit_account": 9,
    "security_name": 3,
    "amount": 7
  },
  "conditions": {
    "debit_account_starts_with": "58.01",
    "credit_account_starts_with": "76"
  },
  "multi_row": {
    "indicator_field": 5,
    "indicator_value": "Кол.",
    "quantity_column": 7
  },
  "operation_type": "Покупка"
}
```

#### Создание правил

Правила можно создать двумя способами:
1. **Вручную** — через форму в интерфейсе
2. **Через AI** — пользователь описывает задачу словами (например: «Найди все покупки по дебету 58.01, кредиту 76»), AI анализирует файл и генерирует JSON-правило

### 4.4. AI-ассистент (AI Martin)

Центральный элемент системы. Чат-интерфейс, через который аналитик управляет всем процессом.

#### Что умеет AI Martin:

| Действие | Описание |
|----------|----------|
| **Анализ файла** | Пользователь загружает файл → AI смотрит структуру, описывает что в нём |
| **Генерация правила парсинга** | Пользователь описывает задачу → AI создаёт JSON-правило |
| **Генерация правила аудита** | Пользователь говорит «сравни A и B по таким-то полям» → AI создаёт правило сравнения |
| **Запуск парсинга** | AI запускает парсинг по созданному правилу, показывает результат |
| **Запуск аудита** | AI запускает сверку, показывает расхождения |
| **Ответы по данным** | AI отвечает на вопросы: «сколько сделок покупки?», «какая общая сумма?» |
| **Подготовка отчёта** | AI формирует сводку по результатам аудита |

#### Как работает под капотом:

1. Пользователь пишет сообщение
2. Бэкенд формирует контекст для LLM:
   - Описание проекта
   - Список источников и их структура
   - Превью данных (первые N строк)
   - Доступные инструменты (tool calling)
3. LLM отвечает текстом и/или вызывает инструменты
4. Результаты инструментов показываются пользователю inline в чате

#### Примеры диалогов:

**Парсинг:**
```
Пользователь: Загружаю карточку счёта. Мне нужны строки где дебет 58.01, кредит 76.
[прикрепляет файл]

AI Martin: Я проанализировал файл. Вижу 2400 строк, шапка из 7 строк.
Вот сгенерированное правило:
{ "conditions": { "debit": "58.01", "credit": "76" } }
Найдено 156 записей. Показать превью?
```

**Аудит:**
```
Пользователь: Сравни данные УК и Брокера. Они должны совпадать по дате регистрации, 
бумаге и сумме.

AI Martin: Создаю правило сверки. Ключи: дата_рег + рег_номер/ISIN + тип_операции.
Сравниваю суммы с допуском ±1 руб.
Результат: 148 из 156 совпали. 8 расхождений. Показать?
```

**Подключение БД:**
```
Пользователь: Подключись к моей базе PostgreSQL: host 192.168.1.10, 
база operations, таблица deals.

AI Martin: Подключился. Таблица deals: 12 колонок, 5430 записей.
Колонки: id, trade_date, security, isin, qty, amount, ...
Импортировать в проект?
```

### 4.5. Движок аудита

Сравнение двух (или более) источников данных по настраиваемым правилам.

#### Правило аудита описывает:

| Параметр | Описание |
|----------|----------|
| Источник A | Какой набор данных (например, «УК») |
| Источник B | С чем сравниваем (например, «Брокер») |
| Ключи сопоставления | По каким полям искать пару (дата, бумага, тип операции) |
| Поля сравнения | Что именно сверяем (сумма, количество) |
| Допуски | Разрешённое расхождение (например ±1 руб. по сумме) |
| Агрегация | Нужно ли группировать записи перед сравнением (как в случае с ДЕПО) |

#### Результат аудита:

Таблица с колонками:
- Данные из источника A
- Статус сопоставления с каждым источником B (✅ / ❌)
- Детали: найденная пара, расхождение
- Фильтры: «Все / Только проблемы / Только ок»
- Экспорт в CSV / Excel

## 6. API (основные эндпоинты)

```
GET    /api/projects           — список проектов

```

```
POST   /api/projects/:id/upload          — загрузить файл(ы)

GET    /api/projects/:id/records         — получить записи (фильтры, пагинация)
```

```
GET    /api/projects/:id/parsing-rules   — список правил

POST   /api/projects/:id/parse           — запустить парсинг
```

```
GET    /api/projects/:id/audit-rules     — список правил аудита
POST   /api/projects/:id/audit-rules     — создать правило
POST   /api/projects/:id/audit/run       — запустить аудит
GET    /api/projects/:id/audit/results   — результаты
```

### AI
```
POST   /api/projects/:id/chat            — отправить сообщение AI
GET    /api/projects/:id/chat/history    — история чата
POST   /api/ai/generate-rule             — сгенерировать правило из файла

```

## 7. Интерфейс

### Структура навигации

```
📊 Проекты                          ← выбор / создание проекта
  └── [Текущий проект]
      ├── 💬 AI Martin              ← чат с ассистентом (главный экран)
      ├── 📁 Данные
      │   ├── Загрузка файлов       ← drag & drop, выбор файлов/папок
      │   ├── Подключение БД        ← форма credentials
      │   └── Просмотр              ← таблица + поиск + фильтры + экспорт
      ├── ⚙️ Правила
      │   ├── Парсинг               ← список правил, создание, редактирование
      │   └── Аудит                 ← список правил сверки
      ├── 🔍 Аудит
      │   ├── Запуск                ← выбор правила, запуск, превью
      │   └── Результаты            ← таблица с подсветкой, фильтры, экспорт
      └── 📊 Отчёты                 ← сводка, дашборд, экспорт
⚙️ Настройки                        ← API ключи, параметры LLM
```

### Ключевые экраны

**1. AI Martin (чат) — главный экран**
- Полноэкранный чат
- Можно прикреплять файлы к сообщениям
- Результаты парсинга/аудита показываются inline (таблицы, графики)
- История сохраняется

**2. Загрузка данных**
- Drag & drop зона
- Предпросмотр файла (первые 20 строк, названия колонок)
- Маппинг колонок (автоматический + ручной)
- Режимы: «Заменить» / «Добавить»

**3. Просмотр данных**
- Таблица с пагинацией (по 100 записей)
- Строка поиска (полнотекстовый по всем полям)
- Экспорт в CSV
- Счётчик записей

**4. Результаты аудита**
- Таблица с подсветкой (зелёный — ок, красный — расхождение)
- Фильтры: Все / Проблемы / Совпало
- Детали по каждой строке (что нашлось, что не нашлось)
- Экспорт в CSV / Excel

## 9. Сценарии использования

### Сценарий 1: Аудит сделок ОПИФ (текущий)

1. Создать проект «Аудит ОПИФ Фонд X»
2. Загрузить карточку счёта УК (Excel) → парсинг по правилу «Дт 58.01 / Кт 76»
3. Загрузить отчёты брокера (папка с Excel) → парсинг раздела 1.2
4. Загрузить выписки ДЕПО (PDF) → парсинг Зачисление/Списание ЦБ
5. Запустить аудит → сверка УК↔Брокер, УК↔ДЕПО
6. Посмотреть расхождения, экспортировать в CSV

### Сценарий 2: Произвольная сверка двух Excel-файлов

1. Создать проект «Сверка реестров»
2. Сказать AI Martin: «Загружаю два файла. В первом — реестр сделок. Во втором — выписка банка. Нужно сопоставить по номеру договора и сумме»
3. AI анализирует оба файла, предлагает маппинг колонок
4. AI создаёт правило аудита, запускает сверку
5. Показывает результат: что совпало, что нет

### Сценарий 3: Работа с внешней БД

1. Создать проект «Анализ Base X»
2. Сказать AI: «Подключись к PostgreSQL, host 10.0.0.5, база accounting, таблица journal»
3. AI подключается, показывает схему таблицы
4. Пользователь: «Покажи все записи за март 2025, где сумма больше 1 млн»
5. AI формирует запрос, показывает результат

## 10. Нефункциональные требования

| Параметр | Требование |
|----------|-----------|
